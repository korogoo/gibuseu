# 자동화 동작 방식

## 회차 / 조 배정 (2026-07-22, 4일 주기로 개편 — `docs/meeting-notes/2026-07-22-4일주기-전환.md` 참고)

- 플로우: D-2 자정까지 주제 제출(이슈 등록) → D-1에 `assign_teams.py`가 제출된 주제의 연관성(소분류 우선, 그다음 대분류) 기준으로 2~3인 조 배정 → D-day 23:00 발표
- **조 배정이 발표 등록보다 나중이다** (예전엔 반대였다 — 랜덤으로 먼저 조를 정하고 등록은 거기 맞춰 라벨링됐음). 그래서 `teams/history.yaml`에 없는 미래 회차 날짜로 이슈를 등록해도 정상이고, 그 시점엔 조 라벨이 안 붙어 있는 게 맞다
- 조 재배정은 GitHub Actions `assign-teams` 워크플로우로 실행해라 (`gh workflow run assign-teams.yml`). 로컬에서 `scripts/assign_teams.py`를 직접 돌리면 저장소 상태와 어긋날 수 있으니, 항상 워크플로우를 통해 실행해라
- 회차 날짜는 기본적으로 "마지막 회차 + 4일"로 자동 계산된다. 주기가 안 맞는 예외적인 회차(예: 2026-07-23 — 3일→4일 전환 회차)에만 `--date`로 직접 지정해라
- `teams/history.yaml`은 `assign_teams.py` 실행 결과로만 갱신되게 유지해라
- 주제 제출 마감(D-2)을 못 지킨 사람은 자동으로 가장 작은 조에 강제 편입된다 (연관성보다 8명 전원 참여 원칙이 우선)
- 조 편성 디스코드 공지에는 조별 공통분야(`common_field_label`), "왜 묶었는지" 한 줄 설명, 조별 발표 주제 링크가 들어간다
- **조 배정 자체는 `OPENAI_API_KEY` 시크릿이 있으면 OpenAI(`gpt-4o-mini`, `llm_group_and_explain`)가 완료기준 원문까지 읽고 나누고, 그 조 편성 이유도 같이 써준다.** 항상 정확히 3개 팀(크기 2~3, 8명이면 3/3/2)으로 나누게 프롬프트/검증에서 강제한다 — 팀 개수가 유동적이면(예: 2/2/2/2) 존재하지 않는 "N조" 라벨이 필요해질 수 있어서다. 시크릿이 없거나, `openai` 패키지가 없거나, API 호출이 실패하거나, 응답의 인원 구성이 실제 제출자와 안 맞으면(이름 누락/중복/팀 개수·크기 불일치 등) 조용히 규칙 기반(`group_by_relevance`+`explain_team`)으로 대체한다 — 외부 API 장애가 회차 배정 자체를 막으면 안 되기 때문
- **이유 문장은 "둘 다 OO 분야다" 식으로 대분류만 언급하는 뭉뚱그린 설명이 아니라, 완료기준에 나온 구체적 개념·메커니즘이 어떻게 이어지는지 짚게 프롬프트에 명시했다** (예: "트랜잭션 격리 수준이 스프링에서 AOP 프록시로 실현된다"). 진짜 연결고리가 없으면 억지로 만들지 말고 그 한계를 인정하게 시킨다 — 새 프롬프트 문구를 고칠 때 이 기준을 유지해라. 규칙 기반일 때의 "왜 묶었는지"는 같은 소분류 > 같은 대분류 > 완료기준 텍스트에서 뽑은 키워드 겹침(코퍼스 전체에서 흔한 단어는 `STOPWORDS`/문서빈도 필터로 제외) > 그냥 "카테고리가 인접해서" 순으로 설명을 고른다. 이 필터가 부실하면 "있다/설명할" 같은 서술어가 키워드로 잘못 뽑힐 수 있으니, 이상한 키워드가 보이면 `STOPWORDS`에 추가해라
- 조 개수가 3보다 커질 일은 이제 없지만(항상 정확히 3개), 라벨이 없으면 `assign_teams.py`가 죽던 문제가 있었어서 `ensure_team_labels_exist()`가 배정 전에 없는 "N조" 라벨을 자동 생성해둔다 — 안전장치로 남겨뒀다
- 임의의 텍스트를 자동화 스크립트 없이 디스코드로 보내야 할 때(조 배정 수동 보정 등)는 `gh workflow run manual-announce.yml -f message="..."`를 써라
- `members.yaml`에는 항상 정확히 8명을 채워둬라 (모자라면 `assign_teams.py`가 실패한다). 세션 시작 시 이 파일이 비어있거나 8명이 아니면 먼저 사용자에게 확인해라

## 발표 등록 검증

- 형식 검사는 `validate-presentation` 워크플로우(`scripts/validate_presentation.py`, 봇 이름 "기부스지키미👀💸")가 자동으로 처리한다 — 결과 코멘트를 신뢰하고 그대로 안내해라
- 검사 항목: 발표자 이름이 `members.yaml`에 있는지, 발표일이 `YYYY-MM-DD 23:00`인지(시간은 고정이지만 표기는 필수 — 별도 시간 필드는 없고 발표일에 같이 적는다), 완료기준이 `-`로 시작하는 줄 2개 이상인지, 소분류 "기타" 선택 시 직접입력이 채워졌는지, 블로그 링크가 URL 형식인지
- 회차 매칭(조 배정, README 표, 리마인더)은 발표일 필드에서 날짜 부분만 떼어(`lib.py`의 `date_part`) 비교한다 — "YYYY-MM-DD 23:00"이든 예전 형식 "YYYY-MM-DD"든 둘 다 매칭된다
- 조 라벨(`1조`/`2조`/`3조`)은 여기서 붙이지 않는다 — `assign_teams.py`가 D-1에 조를 배정하면서 붙인다. 등록 시점엔 아직 그 회차 조가 안 정해졌을 수 있어서 정상이다
- 문제 없이 등록된 이슈는 디스코드에 "📢 새 발표 등록!"으로 주제/분야/발표자/일시를 공지한다 (`state/announced.json`으로 중복 방지, 이슈당 최초 1회). 이미 공지된 이슈가 수정(edited)되면 "✏️ 발표 정보 수정됨" 알림에 필드별 "기존 → 변경" diff를 붙여서 보낸다 — `github.event.changes.body.from`(수정 전 본문)을 워크플로우가 `PREVIOUS_BODY` env로 넘겨주고, `build_diff()`가 발표자/발표일/소분류/유형/블로그 링크를 비교한다. 학습 완료기준은 길어서 값은 안 보여주고 변경 여부만 알린다. `PREVIOUS_BODY`가 없으면(예: title만 바뀐 경우) diff 없이 기본 문구만 나간다

## 카테고리 수정

- 대분류/소분류를 바꿀 때는 `scripts/generate_issue_templates.py`의 `CATEGORIES`를 고치고 스크립트를 다시 실행해라. `CATEGORIES.md`도 같이 갱신해라

## 알림 (기술부채상환 컨셉으로 유쾌한 톤 유지)

- `assign-teams`: 조 배정이 실제로 일어난 날(다음 회차의 D-1)에만 디스코드로 공지해라 (스킵된 날은 조용히 지나가게 둬라). 문구: "이번 회차(N/D(요일)) 조 편성 나왔습니다 😘🫰💸"
- `remind-missing`: 매일 09시/21시에 실행되지만 "다음 발표일의 D-2(주제 제출 마감일)"에만 실제로 알림이 나가게 되어 있다 (하루 2회). 문구는 "💸 오늘 자정까지 주제 제출 마감! 아직 안 낸 사람" 톤 유지
- `remind-overdue`: 매일 09시, 이슈가 닫힐 때까지 매일 반복 알림이 나가게 되어 있다 (의도적으로 중복 방지 없음 — 강제성 확보 목적). 고정 문구: "취업은 조상님이 시켜주나요? 미루지 말고 블로그 작성하세요👴🏻💸"
- `remind-one-hour`: 15분 간격 스케줄 + Issue opened/edited 이벤트, 두 트리거로 실행된다. 발표 시간이 23:00 고정이라 각 Issue의 발표일에 23:00을 붙여 목표 시각(KST, `zoneinfo` 사용)을 계산하고, "목표 시각 - 1시간"을 넘긴 걸 발견하면 그때 한 번만 알린다. `state/reminded_one_hour.json`으로 중복 방지 (라벨 아님 — 이슈에 내부용 라벨이 보이는 걸 원치 않는다는 운영 방침). GitHub Actions는 "미래 특정 시각에 실행"하는 진짜 예약 기능이 없어서 폴링 + 이벤트 트리거 조합이 최선이다 (등록 직후 목표시각이 이미 1시간 이내로 임박한 경우를 이벤트 트리거가 즉시 잡아줌)
- `notify-deleted`: 이슈가 삭제(deleted)되면 "🗑️ 발표 취소/삭제됨" 알림. 삭제된 이슈는 API로 재조회가 안 되니 웹훅 payload(`github.event.issue.*`)를 env로 직접 넘겨 쓴다
- `celebrate-closed`: 이슈가 닫힐(closed) 때 블로그 링크가 채워져 있으면 "🎉 이 귀찮은 걸 해내다니... 멋지십니다... {발표자}" + "📝 블로그 구경가기 → {링크}" 알림. 이어서 다음 회차 주제를 아직 안 낸 사람이 있으면 "{이름, 이름...}당신들은 언제 쓰실 거에요..?"를 같은 메시지에 덧붙인다 — `lib.py`의 `missing_members`(발표일 매칭 기준, 이슈 생성일 기준 아님)를 재사용한다
- 블로그 링크 없이 닫히면 **바로 재오픈 + 안내 코멘트**로 사실상 못 닫게 막는다. GitHub Actions는 닫힘 이벤트를 닫힌 뒤에야 받기 때문에 "닫기 버튼 자체를 막는" 진짜 차단은 불가능해서 이게 최선이다 — 다시 닫으면 이 워크플로우가 또 걸려서 블로그 링크가 채워질 때까지 반복된다
- Discord는 webhook `username` 필드에 `기부스지키미👀` (BOT_NAME), Issue 코멘트는 본문에 `**[기부스지키미👀💸]**` 프리픽스(볼드) (COMMENT_PREFIX) — 둘 다 `scripts/lib.py` 상수를 써라, 표기가 서로 다른 게 의도된 것이다
- 새 알림 문구를 추가할 때는 이 유쾌한 "기술부채 갚기" 톤을 유지해라 (진지한 사무 톤으로 되돌리지 마라)

## Projects 보드 (제거됨)

한때 Projects v2 보드(#8)와 `add-to-project`/`scripts/sync_project.py`, `PROJECT_TOKEN` 시크릿으로 발표일을 동기화했는데,
"캘린더 뷰가 아니면 필요 없다"는 유저 판단으로 보드·워크플로우·시크릿을 전부 삭제했다.
일정 확인은 README의 "발표 진행상황" 표와 디스코드 공지로 충분하다고 결론남 — 다시 만들자는 요청이 없으면 재도입하지 마라.

## 라벨

- '발표' 라벨은 삭제했다. `blank_issues_enabled: false`로 모든 이슈가 발표 등록 템플릿을 거치게 했으므로, `gh issue list`는 라벨 없이 전체로 조회해라
- '1시간전알림' 라벨도 삭제했다. 이슈에 사람이 볼 필요 없는 내부용 라벨을 남기지 않는 게 운영 방침 — 새로 중복방지 로직을 추가할 때 라벨이 아니라 `state/*.json` 파일을 써라

## 상태 파일 (state/)

- `state/announced.json`, `state/reminded_one_hour.json` — 이슈 번호 배열을 담은 JSON. `lib.py`의 `load_state_set`/`save_state_set`으로 읽고 쓴다
- 이 파일들을 쓰는 워크플로우(`validate-presentation`, `remind-one-hour`)는 `contents: write` 권한과 커밋 스텝이 필요하다. 같은 이슈 이벤트에 여러 워크플로우(`add-to-project`, `update-readme` 포함)가 동시에 커밋을 시도해서 push가 경합할 수 있으므로, 커밋 스텝은 실패하면 `git fetch` + `git rebase` 후 재시도하는 루프를 넣어라 (기존 커밋 스텝들 참고)

## README 발표 진행상황 표

- `update-readme` 워크플로우가 Issue open/edited/closed/reopened마다 `scripts/update_readme_status.py`를 실행해서 README.md의 `<!-- STATUS:START -->`~`<!-- STATUS:END -->` 구간을 자동으로 다시 쓴다
- 발표일별로 묶어서(소제목 + 표) 렌더링한다. 닉네임/발표 제목/분야·소분야/블로그 링크 컬럼을 쓴다
- 이 구간은 자동 생성이니 사람이 직접 손으로 고치지 말고, 표 형식을 바꾸고 싶으면 `update_readme_status.py`를 수정해라
